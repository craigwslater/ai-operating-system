# Memory Architecture — `~/.claude-local`

This policy defines how the `~/.claude-local` memory and context system is **tiered, budgeted, routed, and maintained**. It is the canonical operational reference for what kind of memory each file holds, when each file loads, how large each is allowed to be, where a correction belongs, and how the maintenance policies that keep all of that true are enforced.

**Provenance.** This document codifies the target architecture designed in the closed research project `claude-local-frontier-v3` — specifically `outputs/target-architecture.md` §3 (tier model), §7 (maintenance plan), and §8 (budgets), which themselves draw on `outputs/diagnostic.md` (the measured current state) and `outputs/frontier-dossier.md` (the CoALA memory taxonomy and the WYCOS-harvested routing/escalation patterns). It is shipped by **Phase 1** of `claude-local-memory-architecture` (2026-05-22). Later phases of that project add operative detail (see §6 status column and §7). Where this document and the frontier-v3 roadmap disagree, the roadmap (`projects/claude-local-memory-architecture/outputs/improvement-roadmap.md`) is canonical for *phase sequencing*; this document is canonical for *the standing policy*.

## 1. Purpose & scope

The diagnostic established the root problem (`diagnostic.md` gap G-4): **no file declares what kind of memory it is**, so episodic session-by-session status narrative drifts into the always-on semantic store, and the unconditional session-start load grows unbounded. The fix is an explicit, enforced model:

- **Every memory-bearing file has a declared tier** that fixes when it loads (§2).
- **Every tier has a token/byte budget**, enforced by measurement, not by line count (§3).
- **Every correction has a declared destination** by type (§4).
- **Every maintenance policy is bound to a trigger with a corrective default that acts**, and reaches at least the hook-detector rung of the escalation ladder (§5, §6).

In scope: the standing policy. Out of scope: the phase-by-phase execution that *moves files into compliance* — that is the `claude-local-memory-architecture` project. This document states the target state and the rules; it does not itself trim any file.

## 2. The five-tier memory model

Every memory-bearing file is assigned a **CoALA type** (how the memory behaves — `frontier-dossier.md` §5) and a **tier** (when it loads). Type and tier are orthogonal: type is semantic/episodic/procedural/working; tier is the load contract.

### 2.1 Tier table

| Tier | Name | CoALA type | Load behavior | Files | Budget (cl100k / ~calibrated) |
|---|---|---|---|---|---|
| **T0** | Procedural always-on | Procedural | Eager — every session, unconditionally | `CLAUDE.md` | ≤3,400 / ~5,660 |
| **T1** | Identity always-on | Semantic | Eager — every session, unconditionally | `MEMORY.md` (shared core) + `MEMORY.local.md` (per-machine overlay) | ≤3,500 combined (core ≤1,000 + overlay ≤2,500) / ~5,800 |
| **T2** | Project working set | Episodic + working | On demand — when a project is named | active `projects/[name]/CONTEXT.md` | ≤1,800 / ~3,000 each |
| **T3** | Reference / procedural detail | Procedural + semantic | On demand — via an explicit pointer | `SKILL.md`, skill references, `policies/`, `docs/`, `templates/` | ≤8,000 / ~13,300 per file |
| **T4** | Episodic archive + dormant | Episodic | Rarely — explicit backward-looking lookup only | `log.md`, closed-project files, skill `ROADMAP.md`, eval archives | compaction thresholds — §3.2 |

**The eager tier is T0 + T1 only.** That sum is the entire unconditional session-start cost. Everything else is pointer-loaded — preserving the just-in-time loading pattern the diagnostic confirmed is already correct for everything except the always-on tier.

### 2.2 Membership rules

Each tier carries a one-line membership rule. A file whose content violates its tier's rule is **drift**, and the §6 detectors fire on it.

- **T0 — `CLAUDE.md`:** procedural memory only — behavioral rules, the priority hierarchy, decision primitives, workflow-protocol pointers. No identity facts, no project status, no narrative. Test (Anthropic, dossier Lens H): *"Would removing this cause Claude to make mistakes?"* If no, it does not belong in T0.
- **T1 — `MEMORY.md` (shared core) + `MEMORY.local.md` (per-machine overlay):** identity-stable semantic memory, split by `cross-device-os-sync` S2a (2026-06-21) and both eager-loaded. The shared core (`MEMORY.md`, tracked by the framework repo) holds who Craig is — contact, professional summary, tools, education. The per-machine overlay (`MEMORY.local.md`, gitignored) holds the Active Projects table as a *pointer registry* plus the e-commerce brand section. No session-by-session status narrative in either. The registry's governing rule: *"This table is a registry, not a status board."*
- **T2 — active `CONTEXT.md`:** episodic + working memory for one live project — current status, next steps, decisions locked, open questions. The handoff a fresh session reads to resume. One project's worth, loaded only when that project is named.
- **T3 — reference detail:** procedural and semantic detail too large or too specialized to be always-on. Loaded on demand via a pointer from T0/T1/T2.
- **T4 — episodic archive + dormant:** the append-only historical record and dormant closed-project material. Loaded only on an explicit backward-looking lookup. Subject to compaction (§3.2, §6 M3/M4).

### 2.3 How type is assigned

Type and tier are assigned **by location and tier rule**, not by editing a type header into all ~550 workspace files — that keeps the blast radius of this model low. A file's path determines its tier (e.g., any `projects/*/log.md` is T4); its tier determines its CoALA type per the table above. The membership rules in §2.2 are the test for whether a file's *content* matches the tier its *location* assigns it.

## 3. Size budgets & the measurement convention

### 3.1 Dual-unit convention and the calibration

All budgets are stated in **both** units:

- **cl100k tokens** — the reproducible, offline count produced by `tiktoken cl100k_base` in `measure-memory.py`. **This is the enforced unit.** Every budget threshold and every gate in §6 is a cl100k number.
- **Calibrated tokens** — an *estimate* of the true Claude-tokenizer cost, computed as cl100k × **1.665**. Advisory context only; never the enforced figure.

The ×1.665 calibration was re-derived in Phase 1 against three independent ground-truth points spanning distinct corpus types and is confirmed for the prose-class eager-load tier (T0 + T1). Full derivation, data points, and the structured-file caveat (structured T3 files run nearer ×1.62): `projects/claude-local-memory-architecture/outputs/calibration-rederivation-s1.md`.

Budgets are enforced **by token/byte count, never by line count**. This is the `diagnostic.md` C1 lesson: a line budget is trivially gamed by long single-*line* table cells — the exact mechanism by which `MEMORY.md` reached its current state while staying "within" an 80-line budget. `measure-memory.py` is the enforcement instrument; M2 (§6) is the enforcement trigger.

### 3.3 The eager-load ceiling

The unconditional eager-load is the token cost paid at the start of **every** session before any project is named — `CLAUDE.md` + `MEMORY.md` + the per-machine overlay `MEMORY.local.md`, per `policies/session-start.md` steps 1–2. The ceiling is **≤7,000 cl100k**. Rationale: the context-engineering research in `frontier-dossier.md` ties a large standing context to measurable reasoning degradation ("context rot"); the ceiling caps the always-on load at roughly one-third of its current measured size. T0 (≤3,400) + T1 core (≤1,000) + T1 overlay (≤2,500) = 6,900, leaving 100 cl100k of headroom inside the ceiling.

## 4. Correction routing

When a correction is made or a consolidated entry is written, it has a **declared destination by type** (WYCOS-harvested pattern, `frontier-dossier.md` / `target-architecture.md` §7.3). This is what makes every maintenance policy that says "move the detail" unambiguous: the routing table says exactly where the detail goes.

| Correction / content type | Destination | Tier | Why |
|---|---|---|---|
| Behavioral rule, decision primitive, anti-pattern | `CLAUDE.md` | T0 | Procedural memory — governs how Claude acts. |
| Identity fact, stable preference, professional context | `MEMORY.md` | T1 | Identity-stable semantic memory. |
| Project status, next steps, locked decision, open question | that project's `CONTEXT.md` | T2 | Episodic/working memory for the live project. |
| Skill follow-up, planned improvement, deferred item | that skill's `ROADMAP.md` | T4 | The skill's canonical follow-up inventory (`policies/skill-roadmap.md`). |
| Historical detail, session-by-session narrative, superseded content | that project's `log.md` | T4 | Append-only episodic archive. |
| Reusable procedure or reference detail | the relevant `SKILL.md` / reference file or a `policies/` doc | T3 | Procedural/semantic detail, loaded on demand. |

**Routing test.** Before writing a correction, ask: *which type is this?* A status narrative does not belong in `MEMORY.md` (that is the G-1/G-4 drift); a behavioral rule does not belong in a project `CONTEXT.md`; historical detail does not accrete inline in a status cell. If a piece of content seems to belong in two places, it is usually two pieces of content — split it and route each. **Move, do not duplicate:** content lands in exactly one canonical tier; other files point to it.

## 5. The escalation ladder

Gap G-3 is the workspace's proven failure mode: a corrective rule was *detected* over a year of sessions ago and never *enforced*, because it lived only as prose and stalled on a missing confirmation. The standard that prevents a repeat (WYCOS-harvested, `target-architecture.md` §7.2):

> **prose rule → checklist gate → hook detector → mechanical script**

Each rung is a stronger enforcement guarantee than the last. The standard has two binding clauses:

1. **Every maintenance policy (M1–M9) must reach at least the hook-detector rung.** A maintenance rule that exists only as prose in a policy file is — by this workspace's own history — not a rule. Prose and checklist rungs are way-stations, not resting places.
2. **Recurrence forces promotion.** A policy whose detector fires twice without the underlying drift being corrected is auto-escalated one rung. The auto-escalation *trigger* — `scripts/check-escalation.py`, a persistent recurrence ledger (`scripts/escalation-ledger.json`) — is **LIVE as of Phase 6 Session 2**: the SessionEnd hook records each session's findings to the ledger (`--record`, the canonical once-per-session tick), and a finding still present at the end of two consecutive sessions without correction is surfaced as an `ESCALATE:` item naming the policy's current rung and the next one. `/audit` reads the ledger read-only (`--report`). See §6.1 (the detector inventory) and the "M9 + auto-escalation" subsection below.

**Semantic-policy exception (M10).** Clause 1 has exactly one documented exception. **M10** — contradiction resolution (§6) — is a *semantic* policy: detecting that two files assert incompatible facts requires LLM reasoning over file content and cannot be reduced to a mechanical regex or shell detector. The SessionEnd hook is by design a cheap shell script that degrades gracefully (the M2 precedent — a `bytes/4` fallback when `tiktoken` is absent, a silent skip when `python3` is absent); an LLM contradiction sweep cannot be expressed as one and is too costly to run on every session close. M10's enforcement floor is therefore the **`/audit` deep-pass rung**, not the hook rung — a deliberate, bounded exception to clause 1.

The exception is bounded three ways so it is not a general escape hatch. (1) It applies only to a *genuinely semantic* policy — one whose detection is irreducible to a mechanical script; M10 is the only such policy in the M1–M10 register. (2) M10 still reaches the **detector rung** of the ladder: `/audit` is itself an automated detector, and per §6.1 the hook-vs-`/audit` distinction is *continuity, not rung* — M6-DEAD, M7, and M9 already sit at the `/audit` deep-pass rung for the same reason (judgment cannot be mechanized into a per-session hook). The exception lowers M10 below the hook *sub-rung*, not below the detector rung. (3) Clause 2 (recurrence forces promotion) still binds M10, but with an **`/audit`-keyed cadence**: the auto-escalation ledger is ticked once per *session* by the SessionEnd hook (`check-escalation.py --record`), and M10 does not run on that hook, so M10's recurrence cannot be session-keyed — a contradiction surfaced by one `/audit` deep pass and still unresolved at the next consecutive `/audit` deep pass is the M10 recurrence condition. The ledger-schema support for an `/audit`-keyed finding was implemented in Phase 7 Session 2 (`check-escalation.py --audit-record`, the separate region-scoped `scripts/consistency-ledger.json`); the design decision that it must be `/audit`-keyed was Phase 7 Session 1. Mechanism: the **region-scoped `/audit` deep-pass verifier subagent** (§6 M10 row + the "M10 — what is live after Phase 7" subsection; full design `projects/claude-local-memory-architecture/outputs/m10-mechanism-design-s10.md`). M10 went LIVE in Phase 7 Session 2.

**Every corrective default acts.** Detection without an acting corrective policy is insufficient — G-3 proved it. Every policy in §6 is bound to a named trigger and has a corrective default that *acts*. Craig can always override; the *absence* of a confirmation never stalls the fix.

### M8 — what is live after Phase 4

They still reach the detector rung of the escalation ladder: `/audit` *is* an automated detector; the hook-vs-`/audit` distinction is continuity, not rung.
- **The auto-escalation trigger.** `scripts/check-escalation.py` implements the escalation ladder's recurrence clause (§5 clause 2). It maintains a persistent recurrence ledger (`scripts/escalation-ledger.json`): on each tick it fingerprints every structural-drift finding (normalized — lower-cased, backticks stripped, digit-runs masked, so the same drift fingerprints stably across sessions even as its magnitude changes), increments a per-finding consecutive-occurrence count, and drops any ledger entry absent this run (the drift was corrected — the count resets). A finding reaching **2 consecutive occurrences** is emitted as an `ESCALATE:` line naming the policy's current rung and instructing that the drift be corrected this session or the policy's enforcement consciously escalated one rung. **The SessionEnd hook is the canonical ticker** — it runs `check-escalation.py --record` once per session (the natural "fires twice" = end of two distinct sessions cadence); `/audit` reads the ledger read-only (`--report`, `/audit` Step 1.10) and never ticks it. The ESCALATE output is appended to the SessionEnd report but is never itself recorded — only detector findings are.
- **Container note (Non-Negotiable #4).** `check-lifecycle-integrity.py`, `check-escalation.py`, and the `/audit` Steps 1.9 + 1.10 are byte-identical across surfaces; the SessionEnd-hook `check-escalation.py --record` binding ships to Cowork via the `frontier-hooks` plugin (`hooks/package-hooks-plugin.sh`).

### M10 — what is live after Phase 7

M10 went LIVE with Phase 7 Session 2 — the contradiction-resolution policy, and the **last** maintenance policy in the M1–M10 register. With it, **G-14 is closed** and the `claude-local-memory-architecture` arc is complete: every diagnostic gap G-1–G-14 is closed, and consolidation now has all three legs the frontier dossier's Lens G names — deduplication (M5), forgetting (M3/M4/M9), and **conflict resolution (M10)**.

- **M10 is a semantic policy — the detector is a verifier subagent, not a script.** Detecting that two files assert incompatible facts about the same entity requires LLM reasoning over file content; it is irreducible to a regex/shell detector. M10's detector is therefore a **verifier subagent** spawned by `/audit` Step 1.11 on a deep pass. This is why M10 sits at the `/audit` deep-pass rung — the documented semantic-policy exception to the escalation ladder (§5) — and never runs on the SessionEnd hook.
- **Region-scoped, not edge-pair-scoped (the S10 scoping fix).** The roadmap specified M10's scope as "claim-bearing file pairs surfaced by the typed-edge graph." That misses both real G-14 instances: the only edge class connecting arbitrary file pairs — `file —points-to→ file` — is scoped to *cross-container* edges, and both contradictions (the OpenClaw Fusion date; the `craig-profile.md`↔Rule-7 "Agile" inconsistency) are *intra-container*. The fix: scope by bounded **region**. `scripts/consistency-regions.py` enumerates the regions — the root memory+policies cluster, each active project's own live files, each skill's reference set — each capped at a token budget, with cross-container `points-to` targets widened in. `/audit` Step 1.11 spawns one verifier subagent per region (spawn-prompt template: `skills/audit/references/m10-verifier-prompt.md`). A region sweep reads every file in a cluster together, so an intra-cluster contradiction is in one verifier's context. Full design + the rejected hybrid (the claim-register half was rejected as frontier-theater under Non-Negotiable #7 — M5 already designs out copy-drift): `projects/claude-local-memory-architecture/outputs/m10-mechanism-design-s10.md`.
- **Cost is bounded three ways (S11 sub-decisions, tunable like the M4 thresholds).** (1) **Whole-file read**, no claim-extraction pre-step — a region over the `REGION_TOKEN_CAP` (60,000 cl100k) is sub-chunked into sequential whole-file chunks (`label#1`, `#2`, …); claim-extraction is a recorded follow-up, not built (Non-Negotiable #7). (2) A **cost ceiling** — `MAX_REGIONS_PER_PASS` (12) caps verifier spawns per deep pass. (3) **Rotation** — high-priority regions (root + active projects) sweep every deep pass; low-priority regions (skill reference sets) fill the remaining slots round-robin from a persisted cursor (`scripts/consistency-rotation-state.json`), so every region is reached within a bounded number of passes. Lifecycle-aware scope: only **live** material is swept (the M4/M6/M7 precedent — closed/dormant projects and sealed archives are exempt; a closing project's final consistency check is a `/close-project` follow-up).
- **The corrective default acts — surface and route, no auto-pick.** A detected contradiction is flagged with both file+line locations and the two conflicting assertions quoted verbatim, and routed per the §4 correction table. M10 **does not pick a winner**: a contradiction can legitimately be resolved either way (the OpenClaw instance may even be left as-is), so resolution stays a Craig decision. "Acts" means it does not stall on a confirmation (the G-3 failure mode) — it surfaces and routes unconditionally.
- **Recurrence is `/audit`-keyed (the §5 exception clause 4).** `check-escalation.py --audit-record` ticks a **separate** ledger — `scripts/consistency-ledger.json` — keyed to `/audit` deep passes, not sessions, because M10 does not run on the SessionEnd hook. The reconciliation is **region-scoped**: an `--audit-record` tick is told which regions the deep pass swept, and an absent finding from a region that rotation skipped is *retained* (not re-checked ≠ corrected), not dropped. A contradiction present in two consecutive deep passes that sweep its region escalates.
- **First execution — the first contradiction sweep (Phase 7 Session 2).** Deep pass 1 swept **12 regions** — the 8 high-priority regions (root memory+policies; the 6 active projects, with `job-search` sub-chunked into two) plus 4 low-priority skill reference sets, the remaining 10 deferred to later passes by rotation. The verifier subagents surfaced **4 genuine contradictions** and 6 borderline items; 7 regions returned clean. The four: (1) the `CONTEXT.md` size budget — `new-project-intake.md` prescribes a "Hard ≤80-line budget" while `memory-architecture.md` §3.1 says budgets are "enforced by token/byte count, never by line count"; (2) an intra-file stale section in `ai-operating-system-maintenance/CONTEXT.md` (CS#3 drift marked both "resolved" and "real and unfixed"); (3) the Cowork plugin-upload location — `docs/environment-specific-paths.md` says "Organization Settings → Plugins" while `claude-local-frontier/ROADMAP.md` records that path is org-account-only and personal accounts use the Customize tab; (4) a target-company "12-month combination arc" claim in a job-search draft contradicted by that file's own sourcing audit ("10-month arc, rounded up"). All four were recorded to `consistency-ledger.json` (occurrences = 1 — a first pass cannot escalate) and dispositioned with Craig; per M10's surface-and-route default the resolutions are Craig's, not auto-applied. Two of the four (#1 and #2) sit inside the memory architecture's own files — M10 found real drift on its first run.
- **Container note (Non-Negotiable #4).** `consistency-regions.py`, `check-escalation.py --audit-record`, the `/audit` Step 1.11 prose, and the verifier-subagent spawn are byte-identical across surfaces; subagent spawning is available in both Cowork and Claude Code, and no part of M10 depends on a hook — that is the whole point of the §5 exception.
