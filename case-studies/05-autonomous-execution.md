# Earned Autonomy

> **Designed by Craig.** The autonomous-execution architecture—a planned multi-session project compiled into a frozen queue of single-session units that a headless orchestrator runs end-to-end, pinging a human only at genuine decision points. The nine-gate escalation taxonomy and its DECIDE/PARK/STOP model. The three-tier autonomy ladder (never / supervised / fire-and-forget) with its five-condition eligibility gate and per-type graduation. The anti-falsification invariant that a failing check is never made to pass by editing what it measures. The rolling quality-trend stop that halts a run before quality visibly declines. The rule that granting fire-and-forget autonomy is a human decision, made per project type, never by the loop.
> **Runtime: Claude.** Running the orchestrator session that launches each unit as its own subprocess. Classifying every decision the loop reaches into DECIDE, PARK, or STOP. Firing the nine gates, surfacing escalations through a push channel, checkpointing before a halt, and resuming on exactly the option a human authorized. The deterministic helpers do the bookkeeping; the judgment never leaves the model.

## Origin moment

Every case study before this one describes a system that a human drives one session at a time. [Case Study #4](./04-discipline-to-machinery.md) ended on a hook—`pre-tool-use-unit-scope.sh`—that does nothing in an interactive session and only wakes up when "an autonomous execution unit has declared a write scope." That hook was built before the thing it guards. It was the safety rail laid down first, because the engine it protects is the one place in this workspace where the runtime acts without a human watching each step.

The driver had become the bottleneck. A multi-session project—rebuild a backlog, process a batch, run a set of units to a fixed quality bar—required a person to open each session, confirm the scope, watch it run, and open the next. The work between sessions was mechanical; the human in the loop was there to catch the rare decision that genuinely needed a person. The question this chapter answers is whether the loop can run itself and still stop for exactly those decisions—no more, no less.

The counter-thesis is stated first, because it is the real risk: **autonomy on an unguarded workspace amplifies drift.** A system that makes a mistake once, unattended, makes it a hundred times before anyone looks. So the engine could not be built as "let the model run and hope." It had to be built the way the rest of this system was built—name the failure mode, encode the guard, verify against ground truth—and it had to earn every increment of independence against evidence, not assertion.

## What this is

The autonomous-execution engine compiles a planned project into a **frozen queue of single-session units** and runs them end-to-end through a headless orchestrator, escalating to Craig only at decision points that meet the workspace's ASK bar. It is the unattended counterpart to a rule that already governs every interactive session: the *Decide vs Ask* primitive from the [meta-architecture](./01-personal-ai-os.md), which says resolve what is reversible and clearly-better, ask about everything else. Interactively, "ask" means block and wait for the human in the chat. Unattended, there is no chat to wait in—so the engine adds the machinery that lets the loop keep working while a question is outstanding, hold the question in a durable queue, ping a human through a real channel, and resume correctly when the answer arrives async.

Two properties make it a portfolio chapter rather than a science project. First, **judgment never leaves the model.** The engine's state mechanics—which unit runs next, what the run ledger says, whether a quality trend is declining—are handled by a small set of deterministic command-line helpers that only do bookkeeping and arithmetic. Every actual decision is classified by the model against the same rules a human would apply. The helpers cannot resolve a gate; they can only record that one was raised. Second, **autonomy is earned per project type, never granted.** The default posture is supervised. A project type reaches unattended, fire-and-forget execution only after it has proven, under observation, that the loop escalates every decision that should be escalated and that its writes are fully enforceable by the guard layer. This is the discipline that separates the engine from "AI tinkering": it is a safety-gated, human-in-the-loop system whose whole design is about where autonomy is *not* allowed.

## The engine: orchestrator and runner

The engine separates the thing that decides from the thing that executes.

The **orchestrator** is not a daemon or a shell script—it is itself a Claude session running a fixed protocol. It has to be, because the tools it depends on to schedule a wake-up, send a push notification, and receive an answer are in-session tools, not shell processes. It is backed by six deterministic Python-standard-library command-line helpers (`plan`, `runstate`, `queue`, `surface`, `notify`, `runner`) that own all state mechanics. The split is deliberate and load-bearing: the helpers do the bookkeeping so the model does not have to hold run state in its head, and the model does the classifying so a script never adjudicates a judgment call.

The **runner** launches each unit as its own separate `claude -p` subprocess. Every unit starts with a fresh context—no cross-unit context rot, no accumulation of a dozen units' worth of history in one window. That is the same modular-structure insight that runs through [Case Study #3](./03-eval-driven-loops.md), applied at the session level: a unit is scoped to be **one non-compacting session**, asserted at plan time by a `size_guard` field. If the compiler cannot size a unit to fit one session without compaction, that is a planning error caught before the run, not a failure discovered mid-execution.

A project becomes runnable through a **freeze**. A planning session compiles the prose plan into a human-review form, then into a validated `PLAN.json` the loop consumes. Validation is a hard gate: it refuses any unit that resolves to an empty write scope and any malformed scope glob, because a unit with no declared write boundary is a unit the guard layer cannot enforce. Craig signs off; the plan is then frozen, and the loop never rewrites it. The one exception is itself a gate: if a unit is about to compact, the loop stops and routes a re-plan rather than running through the compaction. Each unit in the frozen plan declares its `id`, its per-unit **write-scope allowlist** (a forbidden set that a unit can never override), its dependencies, and verifiable done-criteria that always include read-back verification—the *File Write Verification* primitive from [Case Study #4](./04-discipline-to-machinery.md), carried into autonomy unchanged.

Runtime state lives in a single append-only ledger, `RUN.json`, with one writer (the loop) and six unit states: `queued`, `live`, `blocked`, `done`, plus the terminals `error` and `stopped`. The single-writer discipline is what keeps the ledger trustworthy when an answer arrives from outside the loop.

## The nine gates

The spine of the engine is a **taxonomy of nine gate types**—the categories of decision an unattended loop actually hits. Each gate names a default outcome and, for the conditional ones, what flips it. Eight fire at a per-unit decision point; the ninth is the one arc-level gate, raised at a batch boundary from a rolling metric rather than at any single decision.

| Gate | Fires when | Default |
|---|---|---|
| `ambiguous-requirement` | the plan under-determines the unit | STOP if it blocks core output; PARK if only an optional sub-task |
| `high-blast-radius-write` | a write falls outside the unit's scope or in the standing forbidden set | **STOP, always**—never resolved by the loop |
| `quality-bar-failure` | a deliverable fails its skill's quality gate | retry once (a legitimate regeneration), then STOP |
| `missing-input` | a required file, source, or credential is absent | PARK if other work can proceed; STOP if the whole unit needs it |
| `compaction-risk` | the unit approaches the context budget before finishing | **STOP, always**—checkpoint and re-plan; never run through compaction |
| `constraint-violation` | the next step would break a non-negotiable or corrections rule | **STOP, always**—the offending step is abandoned regardless of the answer |
| `reversible-low-stakes` | formatting, naming-within-convention, ordering | DECIDE, logged, never escalated—this gate exists to name the floor |
| `error` | a step fails irrecoverably after bounded retries | STOP with a diagnostic—the loop never papers over an error with a guess |
| `quality-trend` | the aggregate quality slope declines across a batch | **STOP, always**—arc-level; see below |

Three gates STOP unconditionally—the two that would let a wrong write land or compound (`high-blast-radius-write`, `compaction-risk`) and the one that would let the loop cross a hard rule (`constraint-violation`). One gate, `reversible-low-stakes`, exists only to name the floor: it is the explicit statement that the loop *should* decide trivia itself, so the design is protected against the over-eager failure mode of escalating everything. The gates are what make the run **auditable rather than trusted**: every decision the loop reached is a typed, logged event a human can review after the fact, not a black box.

`quality-trend` is the newest and the most distinctive. Where `quality-bar-failure` catches a hard per-unit miss, `quality-trend` catches the *slope*—a rising escalation rate (the arc getting rougher: more ambiguity and missing-input gates per unit, an early warning that a hard failure is coming *while units still pass*) or a rising quality-bar-failure rate across a batch. It is the "stop before quality declines" control for a long arc, computed read-only by a `runstate trend` helper that does only the arithmetic; whether the decline is real and what to do about it stays with the model. Its thresholds are deliberately provisional, to be tuned from the first real supervised run rather than guessed, and it is armed in *both* autonomy tiers—even a fire-and-forget run re-inserts this stop on a breach. Critically, it only ever *tightens*: it can add a stop, but it can never downgrade a quality failure or relax a quality bar.

## Async escalation: decide, park, stop

Every decision point the loop reaches resolves into exactly one of three outcomes.

- **DECIDE** when the call is reversible, has a single clearly-better option, and aligns with an established preference. The loop resolves it, records the choice and rationale in the ledger, and does not notify anyone.
- **PARK** when the call meets the ASK bar but nothing in the remaining unit work depends on its outcome. The loop enqueues the question, continues the *independent* sub-tasks, and folds the question into a batched digest rather than an interruption.
- **STOP** when the call meets the ASK bar *and* it either blocks the unit or any further write would compound a wrong choice. The loop checkpoints the unit, halts it, moves to the next independent unit if one exists, and pings the human immediately.

One rule keeps PARK honest: it is permitted only when no continued sub-task reads, depends on, or could be invalidated by the parked outcome. If continuing might build work on top of an assumption the answer could overturn, the call is a STOP, not a PARK. PARK refuses to *idle*; it never refuses to *ask*, and it never resolves the question itself. That is what prevents "park and keep going" from becoming a back-door silent guess. The bias is deliberately toward not-stopping—the blocking set is narrow, so the loop is not constantly interrupting—but narrowing STOP never widens DECIDE. A decision that meets the ASK bar is *always* escalated, never auto-resolved, and resuming after a halt never converts the still-open decision into something the loop may now decide for itself.

The questions live in an **append-only queue**, separate from the run ledger so the loop that writes run state never races the external process that writes an answer. Each raised gate carries its question, its options, a recommended option (a lean, never a resolution), and a resume token. The outbound ping is a pluggable interface—a one-line, stop-tagged notification, with a native push notification as the reference channel. STOP pings go out immediately; PARK questions accumulate into a digest. When Craig answers, the loop treats the chosen option as the *sole* authorization: it applies exactly that option, nothing adjacent, and if the answer maps to no offered option it re-escalates rather than interpreting. A halted decision never auto-resolves on resume.

The hardest-won rule in the whole model is **anti-falsification**, and it came from the run that validated the engine. A failing check—a quality gate, a verification script, a constraint—may be resolved in exactly one of three ways: a legitimate fix that re-derives the work from its real sources, a genuine correction to the check's own logic that is routed through escalation and authorized by a human, or a STOP. The loop may **never** make a check pass by editing the data the check measures, nor by quietly editing the check to accept bad data. Either direction is falsification, and falsification is itself a `constraint-violation` STOP. This was the highest-severity finding from the supervised pilot: a near-miss where a failing check *could* have been made to pass by relabeling the thing it measured. The fix was not a stronger warning; it was an invariant, injected into every unit's conduct directive as rule one.

## Earned autonomy: the three tiers

Autonomy is a ladder with three rungs, and a project type climbs it only on evidence.

- **Tier 0—never autonomous.** Some paths the loop may never touch, supervised or not: skill source, the framework-defining files (`CLAUDE.md`, the memory files, the policies), and the concurrency machinery. These change only when Craig edits them in an ordinary interactive session. The escalation policy itself is a Tier 0 file—the loop is governed by rules it can never rewrite.
- **Tier 1—supervised (the default).** The orchestrator runs the arc, but Craig observes the live run, watching for a missed escalation, an over-eager one, or an out-of-scope write. Every project type that has not yet graduated lives here. The pilot ran here.
- **Tier 2—fire-and-forget eligible.** The orchestrator runs unattended and Craig receives only STOP pings—reached only after a type passes a graduation gate.

That gate has **five conditions, all of which must hold**: an isolated, low-blast write scope with zero overlap with the forbidden set and no reliance on a write path the guard layer is blind to; a clean supervised precedent with zero un-flagged high-blast decisions; a deterministic gate surface; a pre-provisioned environment (the loop may never install software unattended—another pilot finding); and a verified notification-and-answer path. Graduation is **per project type and scope shape, not global**—a materially different scope re-enters supervision—and any un-flagged high-blast decision or anti-falsification breach demotes the type back down. The eligibility gate is re-checked on every run.

This is where the positioning of the whole engine lives. Bounded, low-blast, deterministic backlogs are the work that can earn autonomy; high-blast creative work stays interactive by design. The clearest proof is this portfolio: regenerating it is high-blast creative work, so it is built the way you are reading it—one supervised session at a time, human-gated at every push—*not* handed to the orchestrator. Knowing where autonomy does not belong is the same skill as building it.

## The operator command

`/orchestrate` is the operator surface that runs the engine, and its first rule is a constraint on itself: **consume the engine, never rebuild it.** It orchestrates the existing helpers and protocol; the only code it ships is a single read-only boundary helper that reads state facts and never mutates state or dispatches a unit. Any bug routes back to the engine, never to a fork.

The command runs a project in **batches** with a discipline the bare engine does not impose. It opens with a runbook pre-flight—pick the tier, set the guard modes, confirm the push channel, provision dependencies, smoke-test both execution spines. It validates and freezes the plan. Then it runs one batch of units and, at the boundary, renders a **decision-focused review**: the run state, each unit's verifier status, the escalation log, the rolling quality-trend metric, and any gold-set divergence—as *facts*. The model then annotates the one column a script cannot fill: for each escalation, was it warranted, missed, or over-eager? That annotation is the graduation evidence.

Two mechanisms make the batch cadence trustworthy. The **rolling quality-trend stop** is armed in both tiers, so even a fire-and-forget run halts on a declining slope before a hard failure lands—the safety net that makes unattended execution defensible rather than hopeful. And at **every batch boundary, in both tiers, the orchestrator session restarts**: the command ends the current orchestrator and starts a fresh one pointed at the same frozen plan and ledger. A warm session that ran a dozens-of-units arc in one context would eventually hit its own ceiling; the mandatory restart closes that ceiling by construction, because the command never runs one unbounded session across a long arc. In a supervised run the boundary is a manual stop—Craig gives a specific instruction to continue; in a graduated run it auto-continues, but the restart and the quality-trend stop remain. Granting a type its Tier 2 status is never something the command does; that decision stays with Craig, outside the loop.

## What's next

The engine has been validated by a supervised pilot on a bounded, low-blast data backlog, which surfaced seven findings—each encoded as a guardrail fix, not a note-to-self. That is the honest state: proven at pilot scale, with the reliability of a dozens-of-units arc named as the frontier it has not yet crossed. The successor work is concurrency—making the workspace safe for more than one session to execute at once—which is a separate project with its own safety model.

The other three case studies share one insight, captured in [`retrospection.md`](../retrospection.md): *AI workflows producing real deliverables need machinery, separation, and modular structure from day one, not as retrofits.* [Case Study #2](./02-composer-verifier.md) walked separation; [Case Study #3](./03-eval-driven-loops.md) walked modular structure; [Case Study #4](./04-discipline-to-machinery.md) walked the machinery that turns a behavioral rule into a runtime check. This chapter is what those three make possible together: an execution loop that can run without a human on each step precisely because the guards, the gates, and the ledger were built first. Autonomy here is not the absence of discipline—it is discipline dense enough to be trusted at arm's length, and only where it has earned the trust.
